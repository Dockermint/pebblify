# Release Automation and Artifact Attestation

Starting with v0.4.0, every Pebblify release binary and Docker image is published with SLSA provenance and SBOM attestations. This lets you cryptographically verify that an artifact was built by the official CI pipeline from the published source.

Reference: https://docs.github.com/en/actions/how-tos/secure-your-work/use-artifact-attestations/use-artifact-attestations

## What is published

Each tagged release (`v*`) produces:

| Artifact                  | Platforms                                              |
| :------------------------ | :----------------------------------------------------- |
| `pebblify-linux-amd64`    | Linux x86-64                                           |
| `pebblify-linux-arm64`    | Linux AArch64                                          |
| `pebblify-darwin-amd64`   | macOS x86-64 (new in v0.4.0)                          |
| `pebblify-darwin-arm64`   | macOS Apple Silicon (new in v0.4.0)                   |
| Docker image              | `ghcr.io/dockermint/pebblify:<tag>` (linux/amd64, linux/arm64) |

All binaries are built with `CGO_ENABLED=0` from `ubuntu-latest` runners. No host toolchain is required for the target architecture.

## What attestations are generated

For each release:

- **SLSA Build Provenance** — generated by `actions/attest-build-provenance`. Links the artifact to the specific workflow run, git commit, and repository that produced it.
- **SBOM** — generated by `actions/attest-sbom`. Lists all software components included in the artifact in either SPDX or CycloneDX JSON format.

Attestations are stored in the GitHub Attestations registry and are publicly queryable via `gh attestation verify`.

## Verifying release binaries

Install the GitHub CLI (`gh`) version 2.49 or later, then run:

```bash
# Verify a Linux binary
gh attestation verify pebblify-linux-amd64 \
  --repo Dockermint/Pebblify

# Verify a macOS Apple Silicon binary
gh attestation verify pebblify-darwin-arm64 \
  --repo Dockermint/Pebblify

# Verify a Linux arm64 binary
gh attestation verify pebblify-linux-arm64 \
  --repo Dockermint/Pebblify

# Verify a macOS amd64 binary
gh attestation verify pebblify-darwin-amd64 \
  --repo Dockermint/Pebblify
```

A successful verification prints:

```
Loaded digest sha256:<hash> for file://pebblify-linux-amd64
Loaded 1 attestation from GitHub API
✓ Attestation verified — Pebblify/pebblify
  Signed by: https://github.com/Dockermint/Pebblify/.github/workflows/release.yml@refs/tags/v0.4.0
  SLSA level: Build L2
```

If verification fails (tampered binary, wrong repository, or no attestation found), `gh attestation verify` exits non-zero with an error message.

## Verifying the Docker image

```bash
gh attestation verify \
  oci://ghcr.io/dockermint/pebblify:v0.4.0 \
  --repo Dockermint/Pebblify
```

The OCI digest is resolved automatically by the GitHub CLI from the registry.

## How the CI pipeline produces attestations

The release workflow (`.github/workflows/release.yml`) requires these top-level permissions:

```yaml
permissions:
  contents: write
  packages: write
  id-token: write
  attestations: write
```

`id-token: write` allows the runner to obtain an OIDC token from GitHub's identity provider. `attestations: write` allows the runner to store the signed attestation in GitHub's registry.

The pipeline flow for each release tag:

1. `build` job — builds all four binaries with `CGO_ENABLED=0`, uploads them as workflow artifacts.
2. `attest-binaries` job — downloads the four artifacts, calls `actions/attest-build-provenance` and `actions/attest-sbom` with `subject-path: 'pebblify-*'`.
3. `docker` job — builds a multi-arch image (`linux/amd64,linux/arm64`), pushes to GHCR, then calls `actions/attest-build-provenance` with the image digest.
4. `release` job — uploads all four binaries as GitHub Release assets.

Attestation jobs run only on tag pushes. They are never triggered by PRs or branch pushes.

## Security properties

- The signing key is ephemeral and issued by GitHub's OIDC provider for each workflow run. There is no long-lived signing key to protect or rotate.
- The attestation links the artifact digest to the exact workflow file path and git ref that produced it. A binary built outside the official workflow cannot carry a valid attestation for this repository.
- `CGO_ENABLED=0` ensures the binaries contain no host-system library references.
- No secrets are embedded in workflow files. Docker registry login uses `${{ secrets.GITHUB_TOKEN }}` only.

## Downloading and verifying in one step

```bash
# Download the binary for your platform
TAG="v0.4.0"
ARCH="linux-amd64"  # or linux-arm64, darwin-amd64, darwin-arm64

gh release download "$TAG" \
  --repo Dockermint/Pebblify \
  --pattern "pebblify-${ARCH}"

chmod +x "pebblify-${ARCH}"

# Verify before running
gh attestation verify "pebblify-${ARCH}" \
  --repo Dockermint/Pebblify

# Run
./pebblify-${ARCH} --version
```
